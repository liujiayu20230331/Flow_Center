# 企业级 Java 开发规范（Enterprise Java Rule.md）

版本：v1.0  
编码标准：UTF-8  
语言标准：简体中文  

---

# 一、基础规则

## 1.1 编码规则

1. 所有源代码文件必须使用 UTF-8 编码。
2. 所有 Markdown 文档必须使用 UTF-8 编码。
3. 数据库字符集统一使用 utf8mb4。
4. 严禁使用 GBK、ANSI、ISO-8859-1 等编码。
5. 禁止混合编码文件。

---

## 1.2 语言规则

1. 所有代码注释必须使用简体中文。
2. 类名、方法名必须使用英文。
3. 禁止拼音命名。
4. 禁止无意义命名（如 a、b、temp）。

---

# 二、分层架构规范

## 2.1 标准分层结构
com.company.project
├── controller
├── service
├── service.impl
├── repository
├── domain
├── dto
├── vo
├── config
├── exception
├── util

---

## 2.2 Controller 层规范

1. 只负责接收请求与返回响应。
2. 不写业务逻辑。
3. 不操作数据库。
4. 必须使用 DTO 作为入参。
5. 必须使用统一返回结构。
6. 禁止直接返回实体类。

---

## 2.3 Service 层规范

1. 负责核心业务逻辑。
2. 必须使用接口 + 实现类分离。
3. 事务必须定义在 Service 层。
4. 禁止跨层调用 Repository。

---

## 2.4 Repository 层规范

1. 只负责数据持久化。
2. 不允许写业务逻辑。
3. 禁止拼接 SQL 字符串。
4. 必须使用参数化查询。

---

# 三、命名规范

## 3.1 类命名

- XxxController
- XxxService
- XxxServiceImpl
- XxxRepository
- XxxConfig
- XxxException

---

## 3.2 方法命名

- saveXxx
- updateXxx
- deleteXxx
- queryXxx
- handleXxx

禁止：

- doSomething
- testMethod
- aaa()

---

# 四、代码质量规范

## 4.1 方法复杂度

1. 单个方法不超过 50 行。
2. 不允许超过 3 层嵌套。
3. 复杂逻辑必须拆分为私有方法。
4. 禁止超长 if-else 链。

---

## 4.2 常量规范

1. 禁止魔法值。
2. 所有状态码必须定义常量。
3. 禁止硬编码字符串。

---

## 4.3 异常规范

1. 必须定义统一业务异常类。
2. 禁止空 catch。
3. 禁止吞异常。
4. 必须记录错误日志。
5. Controller 层统一异常处理。

---

## 4.4 日志规范

1. 使用 slf4j。
2. 禁止使用 System.out.println。
3. 禁止打印敏感信息。
4. 错误日志必须包含：
   - 请求标识（traceId）
   - 关键参数
   - 异常堆栈

---

# 五、数据库规范

## 5.1 表设计规范

每张表必须包含：

- id（bigint）
- created_at（datetime）
- updated_at（datetime）

建议：

- 主键使用 bigint
- 索引必须合理设计

---

## 5.2 SQL 规范

1. 禁止 select *。
2. 必须添加必要索引。
3. 更新必须带条件。
4. 删除必须带条件。
5. 禁止全表扫描。

---

# 六、事务规范

1. 事务只能定义在 Service 层。
2. 不允许在 Controller 层使用事务。
3. 不允许滥用 REQUIRES_NEW。
4. 事务方法必须为 public。

---

# 七、接口规范
统一返回结构：
- 
   json
   {
     "code": 0,
     "message": "success",
     "data": {}
   }


# 八、安全规范
1. 所有接口必须校验参数。
2. 禁止信任前端数据。
3. 禁止 SQL 拼接。
4. 必须防止重复提交。
5. 敏感信息必须脱敏处理。

# 九、性能规范
1. 查询必须分页。
2. 单次查询不超过 1000 条。
3. 必须避免 N+1 查询。
4. 大表必须加索引。

# 十、禁止行为（用于约束代码生成）
1. 不得生成未分层结构代码。
2. 不得生成所有逻辑写在一个类中的代码。
3. 不得生成无日志、无异常处理代码。
4. 不得生成硬编码 ID。
5. 不得生成 GBK 编码文件。
6. 不得生成演示级代码。
7. 不得擅自引入无关框架。

# 十一、 task.md 记录规则
1. 每次记录之间空一行 
2. 时间格式：yyyy-MM-dd HH:mm:ss 
3. 记录内容包含:
   * 方案决策
   * 接口变更
   * 依赖变更
   * 配置调整
   * 数据库建议

# 十二、注释规范
1. 所有表、表字段必须加上备注
2. 所有方法体加上注释

# 十三、执行沟通规则（项目专用）
1. 本项目进入默认执行模式：已明确的需求和任务，直接实施，不再逐项询问“是否可行”或“是否继续”。
2. 除非用户明确要求先讨论方案，否则默认先落地代码与配置，再反馈结果。
3. 单次任务过程中，禁止重复确认同一事项；若无新约束，按既定 PRD 与 rule.md 持续推进。
4. 若遇到客观阻塞（环境错误、依赖冲突、权限限制），先自行尝试修复；仅在无法继续时一次性说明阻塞点与处理建议。
5. 本规则对本项目后续所有开发任务持续生效。
